# Agent Session Kit v3.0.0

Release date: 2026-03-14

## Summary

ASK 3.0 promotes ASK into a Session OS runtime built on an event ledger, replayable projections, and deterministic command contracts for full-session governance.

This release expands beyond commit/push gates into runtime-native planning, execution, coordination, and delivery control while keeping ASK platform-agnostic.

## Highlights

### 1) Event-Ledger Runtime + Replay Projections

Added and promoted:

- append-only runtime ledger at `.ask/runtime/events.ndjson`
- replay and snapshot model at `.ask/runtime/snapshots/*`
- `ask replay` as a first-class runtime recovery command

Why it matters:

- state is reconstructable and auditable from events
- runtime snapshots are deterministic and testable
- long-running sessions recover faster and with less ambiguity

### 2) Runtime Command Surface for Session OS Operations

Added and stabilized:

- session/policy lane: `session`, `context`, `preflight`, `can-commit`, `pre-commit-check`, `pre-push-check`
- execution lane: `task`, `evidence`, `verify`, `workflow`, `integration`, `integration-auto`
- delivery lane: `feature`, `release`, `promote`, `rollout`, `rollback`

Why it matters:

- one runtime contract from local work to integration-readiness
- fewer script-path drifts and lower policy mismatch risk

### 3) Multi-Agent Coordination Runtime

Added:

- routing/claim/child-session/agent command families
- queue and policy-pack lane for execution-policy behavior
- workflow-provider compatibility checks for adapter-bound integrations

Why it matters:

- clearer task ownership and routing in concurrent agent execution
- deterministic coordination traces for handoff and debugging

### 4) Reliability and Autonomy Upgrades

Added/updated:

- guarded command execution with `180s` stall timeout and one auto-retry
- session diagnostics via `ask session doctor`
- codex context-budget controls via `ask codex context status|ensure|compact`
- phase-based autonomous verify/ship runners

Why it matters:

- stronger resilience during long or unstable local runs
- cleaner autonomous execution with explicit policy gates
